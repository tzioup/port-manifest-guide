Building a Port Manifest

A methodology for cataloging V4 PAI artifacts so V5 can find and port them. Tested on a real migration. Honest about where it breaks. Your PAI does the work — this gives you visibility over the principles so you can spot gaps.

TL;DR

A Port Manifest catalogs every artifact in your V4 PAI installation so V5 can find and port them. Your PAI builds it in two passes: audit the filesystem, then verify the draft against reality. A git diff against your stock baseline only catches modifications to stock files — user-generated content (memory, projects, portal, infrastructure) has no stock counterpart and is invisible to any diff. The manifest process covers both. The manifest is an input to V5 — V5 decides where things go.

To run this: Point your PAI at the starter kit (token-optimized brief, validator, and seed JSONL), not this page. This page is for you to understand the principles and review your PAI's output.

What is a Port Manifest?

A Port Manifest is an LLM-optimized catalog of every artifact in a V4 PAI installation, organized by intent rather than file path. It tells V5: "here's what V4 had, why it existed, what shape it's in, and what depends on what."

It is not an architecture diagram, a dependency graph, or a migration script. It is a navigation aid. V5 reads it and knows where to look — in two tool calls instead of two hundred.

Why intent-based

File paths are meaningless across architecture versions. V4's hooks/lib/btt.ts tells V5 nothing. But "BTT desktop automation bridge — connects Mac BetterTouchTool to VM via autossh tunnel" tells V5 exactly what it needs to decide: port, adapt, or skip.

The manifest has two faces:

  • JSONL — machine-readable, one JSON object per line, schema-validated. This is what V5 queries programmatically.
  • Markdown routing table — human-readable, generated from the JSONL. Category tables, gated items, anti-port list, cross-environment artifacts. This is what you review to spot gaps your PAI missed.
Who does the work

Your PAI builds the manifest. You review it. This guide exists so you understand the principles well enough to catch mistakes — phantom paths, missing directories, V5 prescriptions leaking in, entries grouped too coarsely. The Starter Kit includes an LLM-optimized brief you can paste into a fresh PAI session, plus the validator and template files.

Where This Fits in Migration

V4 Instance → Port Manifest (you are here) → V5 Pulls from Manifest → V5 Self-Places Artifacts

The manifest is one step of the process, not the whole migration. It produces a catalog that V5 consumes. V5 is a different architecture — it decides where things go. The manifest describes V4; it does not prescribe V5.

Critical constraint

The manifest must have no V5 prescriptions. No "this goes in V5's identity folder." No "V5 consolidates these into..." V5 is a different system with different structure. If the manifest tells V5 where to put things, V5 inherits V4's architecture decisions instead of making its own. Every entry has a v5_destination: null field — the null is load-bearing.

Relative paths

All v4_paths inside the PAI repo must be relative to the repo root — hooks/lib/btt.ts, not ~/.claude/hooks/lib/btt.ts. Why: by the time V5 ports from this manifest, V5 will own ~/.claude/ and the V4 backup will live somewhere else (e.g. ~/.claude.v4-backup/). Relative paths resolve against whatever root V5 points at. Paths to artifacts outside the repo (~/work/, ~/portal/, ~/Projects/, ~/.config/) use ~ prefix since those directories don't move during migration.

Diff vs. Manifest — Choosing Your Approach

If your V4 was installed from a shared repo, you might think: "Why not just diff my instance against the stock V4 repo and catalog what's different?" It's a reasonable instinct. Here's how the two approaches compare:

Repo Diff

diff -rq ~/.claude/ /path/to/stock-v4/

  • Fast. Mechanical — minutes, not hours.
  • Exhaustive for modifications. Every changed line in every tracked file surfaces automatically.
  • No human judgment in discovery phase. The diff doesn't miss files it can see.
  • Great for customizations to stock files — custom Algorithm blocks, hook overrides, settings changes.

Port Manifest

Structured audit → intent list → JSONL

  • Captures intent. Groups files by purpose, not by path.
  • Covers user-generated content. Memory, projects, portal, research — things with no stock counterpart.
  • Documents dependencies, debt, and decisions. V5 knows the why, not just the what.
  • Handles off-VM artifacts. Cloud databases, host-machine configs, SaaS state.

Where a diff falls short

The diff catches modifications to stock files but is blind to the rest:

Artifact TypeRepo DiffPort Manifest
Custom Algorithm blocks in stock files Sees every changed line Knows it exists, not the line-level delta
Hook overrides / settings changes Sees the diff Describes the role, not the diff
User memory (hundreds of files) No stock counterpart — invisible Cataloged with counts and role
Projects in ~/work/, ~/Projects/ Outside repo — invisible Cataloged by intent
Portal pages, research, creative output Outside repo — invisible Cataloged with active/dormant status
Cron jobs, systemd services, env vars Not in repo — invisible Enumerated
Cloud state (CF Workers, D1/KV) Not on disk — invisible Listed in cross-environment section
Host-machine state (BTT, LaunchAgents) Not on VM — invisible Listed in cross-environment section
Dependencies between artifacts No model for this Partial (~30% of real deps)
Debt assessment / port readiness No model for this Per-entry assessment

The practical answer: we measured it

We ran git diff against our stock V4 baseline (the initial post-install commit) on our instance. Your numbers will vary depending on how long you've run V4 and how much content you've generated, but the principle held clearly:

What the diff seesOur instance
Modified stock files (customizations to hooks, skills, agents, PAI/, settings)~1%
Added files inside tracked dirs (new hooks, skills, agents, PAI tools)~11%
Files outside tracked dirs (MEMORY, projects, portal, scripts, plugins, infrastructure, research, sessions, commands...)~88%

On our instance, the diff covered roughly 12%. The vast majority of files were user-generated content with no stock counterpart — invisible to any diff. The principle: a diff catches customizations to stock files (Algorithm blocks, hook overrides, agent tweaks), but user-generated content will always be the bulk of what needs porting, and it has nothing to diff against.

Best of both

If you have a stock baseline (your initial install commit, or a fork of the V4 repo), use the diff as input to Pass 1. It surfaces customizations to stock files fast. Then do the filesystem walk for everything the diff can't see. The diff accelerates discovery; the manifest provides the structure V5 needs to consume everything.

Finding your baseline

Your V4 instance is a git repo. The stock baseline is likely your earliest commit — run git log --oneline --reverse | head -1 to find it. If your repo history was squashed or you don't have an initial post-install snapshot, there's nothing to diff against and the manifest process is the only path.

The upstream PAI repo (danielmiessler/PAI) was squashed to a single commit for V5 — you can't clone it and diff against a V4 tree. Your local git history is the only source for a stock baseline.

The 2-Pass Process

The manifest is built in two passes. The first pass discovers what exists and builds a draft. The second pass verifies the draft against the actual filesystem and fixes what the first pass got wrong.

Why two passes? Because the first pass will be incomplete. It always is. You'll miss directories, trust stale counts, overlook root-level orphans, and group things too coarsely. The second pass exists to catch this — the discipline is doing it rather than shipping the draft.

This won't fit in one session

A mature V4 instance has thousands of files across dozens of directories. The filesystem audit alone can exhaust a context window. Expect this work to span multiple sessions and/or parallel agents.

The first session should produce a Spine — a continuation document written to disk that ensures lossless handover. It should contain: filesystem audit results (verified counts, expensive to re-derive), the intent list (every artifact's planned fate), decisions made (grouping, splits, category assignments), and what's done vs. pending. Each subsequent session reads the Spine first, does its work, and updates it before closing.

Without a Spine, each new session re-discovers what the last one already found, makes different grouping decisions, and produces inconsistent entries. The manifest is the deliverable; the Spine is how you get there across context boundaries.

Pass 1: Audit and Draft

1 Walk the filesystem

Before writing a single manifest entry, know what's on disk. find and du are more honest than any index or session log.

  • Count everything: hooks, skills, agents, memory subdirs, portal pages, project dirs, cron jobs, systemd services.
  • Check outside the main directory — project repos, portal content, exchange directories.
  • Walk the repository root explicitly. One-off scripts, research repos, and parked code sitting at the root get missed by every other audit pattern.
  • Identify off-machine artifacts: cloud state, host-machine configs, external services.
  • If you have a stock V4 repo to diff against, run the diff here and use it to seed your customization list.
2 Build the intent list

Before writing JSON, write a plain table: one line per intent, with category, status, and source.

| # | intent          | category       | status  | source          |
|---|-----------------|----------------|---------|-----------------|
| 1 | claude-md       | core-identity  | pending | filesystem audit |
| 2 | hooks-shared    | hooks          | pending | filesystem audit |
| 3 | old-experiment  | projects       | skipped | filesystem audit |

Every V4 artifact must have exactly one fate:

  • Entry — gets a manifest line with full schema
  • Anti-port — explicitly documented as "not porting, here's why"
  • Folded — covered by another entry

If something doesn't appear in any of these buckets, you have a coverage gap. The intent list forces this accounting before you commit to JSON.

Split aggressively up front. Three entries that could have been one is cheap. One entry that later needs splitting into four rewrites paths, dependencies, and port_notes for every affected entry.

3 Write the JSONL

One JSON object per line, 23 fields per entry (see Schema). Key fields:

FieldWhat It DoesCommon Mistakes
intentKebab-case identifier. V5 queries by this.Too granular (one per file) or too coarse (one per category)
v4_pathsArray of paths belonging to this intent.Phantom paths — files that don't exist
port_notesV4 role, ≥80 chars. For someone who knows V5 but has never seen V4.V5 prescriptions leaking in. Thin stubs.
depends_onIntents that must be ported first.Missing or inverted edges, dangling references
debt_assessmentclean|refactor|debt|dropEverything marked "clean" because it feels clean
v5_destinationAlways null.Filling it in. Don't. V5 decides.

Grouping principle: Group by intent, not by path. Three files serving the same purpose get one entry.

Target count: 85–150 entries for a mature installation. Under 85 = catch-all buckets that sacrifice routing accuracy. Over 150 = approaching one-per-file granularity, defeating the compression purpose.

Your PAI should use a TypeScript generator script to produce the JSONL rather than writing raw JSON inline. JSON.stringify() handles escaping correctly; inline JSON in bash is a minefield. Have it run the validator after every batch of ~10 entries to catch errors early.

4 Run the validator

At this point you have a draft manifest. Run the validator (see below). It will likely pass on schema — and that's the trap. Schema-valid does not mean correct. Proceed to Pass 2.

Pass 2: Verify and Harden

Pass 1 produces a draft that's structurally valid but probably wrong in ways the validator can't catch. Pass 2 fixes it.

5 Verify paths against filesystem

The single highest-value check: stat every path in every v4_paths array. If you only do one thing in Pass 2, do this.

Phantom paths are the #1 data quality issue. Files get moved, renamed, or deleted. Hooks that used to live at hooks/X.ts now live at PAI/Tools/X.ts. Without path checking, these phantoms survive indefinitely because the validator can't catch what it doesn't look for.

6 Check coverage gaps
  • Compare your intent list against the filesystem walk from Step 1. Did any directory or file family slip through without an entry or anti-port decision?
  • Check root-level items specifically — files and directories sitting directly at the repository root are the most commonly missed.
  • Verify counts in port_notes against actual find output. Stale counts from indexes propagate silently into port_notes.
7 Check structural integrity
  • Dependency cycles: Run cycle detection on the depends_on graph.
  • Bidirectional consistency: If A blocks B, does B depend on A?
  • Orphan entries: Entries with no depends_on and no blocks. Some are genuinely isolated (projects, portal pages). Others are missing edges (infrastructure entries almost always have real dependencies).
  • Overlapping paths: Two entries claiming the same directory creates routing ambiguity for V5.
8 Generate the routing table and diff

Generate the markdown from the JSONL — don't maintain it separately. It should include:

  • Scope Declaration — what the manifest covers and what it doesn't
  • Category tables — intent, status, priority, summary per category
  • Gated items — entries needing a human decision, with the specific question
  • Anti-port list — what you decided not to port and why
  • Cross-environment artifacts — cloud, host-machine, external services
  • Data-flow chains — pipelines crossing multiple entries that the dependency graph can't represent
  • Credentials list — API keys, tokens, OAuth — needs manual transfer

If evolving from a previous draft, produce a diff showing entries added, removed, and modified with sources. This proves audit findings were addressed, not just acknowledged.

The Schema

23 required fields per entry:

{
  "intent":           "kebab-case-identifier",
  "category":         "core-identity|algorithm|hooks|skills|agents|memory|
                       tools|projects|portal|infrastructure|knowledge|
                       learning|relationships",
  "display_name":     "Human-Readable Name",
  "summary":          "One-line description",
  "status":           "pending|in-progress|ported|skipped|gated",
  "status_updated":   "YYYY-MM-DD",
  "priority":         "P0|P1|P2|P3",
  "effort":           "5min|30min|2hr|4hr+",
  "v4_paths":         ["path/to/file", "path/to/dir/"],
  "v4_role":          "What this does in V4 (longer description)",
  "v4_format":        "File type and structure description",
  "content_nature":   "identity|configuration|data|code|documentation|
                       research|creative",
  "portability":      "portable|adaptable|v4-native",
  "v5_destination":   null,
  "v5_slot_exists":   null,
  "depends_on":       ["other-intent"],
  "blocks":           ["other-intent"],
  "related_indexes":  ["grep command to find related data"],
  "port_notes":       "V4 role description for V5 consumer (≥80 chars)",
  "debt_assessment":  "clean|refactor|debt|drop — explanation",
  "gate":             "Question to answer (if gated) or null",
  "skip_reason":      "Why skipped (if skipped) or null",
  "ported_by_session": null
}

The 13 categories map to the major artifact families in a PAI installation. Your V4 may have different ones — the validator enforces whatever set you define.

The Validator

Write it in TypeScript (Bun). Runnable as:

bun validate-manifest.ts path/to/PORT_MANIFEST.jsonl

It should check:

CheckWhy
All 23 fields presentSchema completeness
Enum values validData quality
v5_destination is nullNo V5 prescriptions
port_notes ≥ 80 charactersNo thin stubs
V5 names / prescriptions not in port_notes (regex)No V5 leaks in prose
Every v4_path exists on diskNo phantom paths
depends_on / blocks reference existing intentsNo dangling references
No dependency cyclesOrdering is possible
No duplicate intentsRouting is unambiguous
Entry count within target rangeNot too coarse, not too granular
Skipped have skip_reason, gated have gate questionNo unexplained gaps

Implementation notes:

  • Path checking: resolve ~ to $HOME, then existsSync(). Run from the repo root so relative paths resolve.
  • Cycle detection: DFS with an in-stack set over depends_on edges.
  • Bidirectional check: warning, not error. Some edges are intentionally one-directional.
  • V5 regex: catch both explicit (V5 expects) and paraphrased (the newer architecture) references. Tune to your V5's naming.

Gotchas

Indexes lie

If your V4 has JSONL indexes or session logs with artifact counts, do not trust them as ground truth. We found 10-50% discrepancies between indexed counts and actual filesystem state. 290 security files the indexes didn't mention. 88 portal directories instead of the indexed 53. Always verify with find.

Phantom paths survive schema validation

Without path existence checking, the validator happily accepts paths to files that were moved or deleted months ago. Add stat checking — it's the single highest-value validator feature. We found 12 phantom paths that survived every other check.

Prose escapes the validator

The validator checks v4_paths for existence and port_notes for V5 prescriptions. But the markdown routing table — data-flow chains, cross-environment lists, credentials tables — is free text no validator touches. We wrote phantom paths into data-flow chain documentation and didn't catch them until manual review. Anything in prose is unguarded.

Root-level orphans

The manifest naturally covers well-known subdirectories (hooks/, skills/, agents/, MEMORY/). But files and directories at the repository root — one-off scripts, research repos, parked code — get missed because no audit pattern looks for them. Walk the root explicitly. We found 10 orphan items with no manifest fate.

Process ceremony without substance

Running a structured methodology (agents, phases, checklists) and then doing the actual work without the structure is worse than not using a methodology at all. It produces a validator-passing artifact that looks rigorous but wasn't built with rigor. If you skip the intent list and go straight to writing JSONL, the resulting manifest will have coverage gaps. The intent list exists to prevent exactly this.

0 validator errors ≠ correct

The validator checks structure. It cannot check completeness — it has no model of what should exist. A manifest with 0 errors and 40% coverage gaps will pass. The only defense is the 2-pass process: Pass 1 drafts, Pass 2 verifies against reality.

Honest Limitations

The manifest has structural limitations that can't be fixed by adding more entries:

LimitationWhyMitigation
Filesystem-only coverage The scanner only sees the local filesystem. Cloud databases, host-machine configs, SaaS state are invisible to find. Cross-environment section documents known off-filesystem artifacts. But "known" is the operative word.
~30% dependency coverage depends_on/blocks captures "port X before Y." It can't represent data-flow chains, runtime ordering, import graphs, or temporal sequencing. Data-flow chains section documents major pipelines in prose. Unvalidated and incomplete.
No completeness detection The validator checks structure, not coverage. 0 errors does not mean 0 gaps. The 2-pass process reduces gaps. "Reduces" is not "eliminates."
Staleness Numeric claims in port_notes drift as the system changes. At active-development pace, counts go stale within a week. V5 should verify counts at port time. The manifest says where to look; V5 counts what it finds.
Catalog, not architecture map The schema can't represent feedback loops (Memory → Hooks → Learning → Algorithm → Memory). depends_on must be acyclic. Accept this. The manifest compresses 200K+ tokens into ~30K of structured navigation. It's compression, not modeling.

What V5 Does With This

The manifest is an input to V5, not a prescription for V5:

  1. Query by intent. V5 reads the JSONL and searches for the artifact family it needs. Two tool calls: find the intent, read the entry.
  2. Read the port_notes. Understands what V4 had, in V4's terms. No V5 assumptions baked in.
  3. Self-place the artifact. V5 decides where it goes in V5's architecture. The manifest provides context; V5 makes the architectural decision.
  4. Check dependencies. Before porting entry X, check if depends_on entries are already ported.
  5. Verify at port time. Don't trust manifest counts or paths blindly. stat the paths. Count the files. The manifest may be stale.
  6. Resolve gates. Gated entries need a human decision before V5 can proceed. The gate question tells V5 what to ask.
The value proposition

Without a manifest, every V5 porting session starts with filesystem archaeology — find, grep, du, reading files to understand what they do, guessing at dependencies. With a manifest, that archaeology is done once and compressed into a queryable format. For dozens of porting sessions ahead, the manifest pays for its 2-3 hour investment many times over.

The Receiving End — V5 Field Notes

Everything above covers building the manifest. This section covers consuming it — what V5 actually encounters when it starts porting from a manifest. Based on a real migration (~2.9GB V4 instance, 408 work sessions, 64 manifest intents). Your numbers will differ; the patterns should generalize.

V5 prompt template

Just as V4 uses BRIEF.md to build the manifest, V5 uses V5_BRIEF.md to consume it. Point your V5 PAI at it when starting the porting process.

Map your retrieval architecture first

The manifest describes V4. V5 needs to know itself before it can route content. Which files load at startup? Which tools search which directories? Which hooks auto-generate content? Without this map, V5 falls into file-name matching — mapping V4 filenames to V5 filenames — which is wrong because the taxonomies are different.

The question to ask

"Where will V5 look for this when it needs it?" — not "what's the V5 equivalent filename?"

Stock vs. flow — the biggest insight

The manifest catalogs everything by intent. But intent doesn't tell V5 whether it needs the content at all. The classification that cuts across every category:

Stock — leave it

Accumulated observations V5 regenerates through its own hooks and learning system.

  • Relationship memory entries
  • Algorithm reflections
  • Session logs / journal entries
  • Learning signals / satisfaction ratings
  • Failure captures

These are calibrated for V4's hooks, schema, and behavior. Porting them contaminates V5's developing model.

Flow — port it

One-time work product V5 cannot recreate.

  • Research investigations
  • Curated discoveries & decisions
  • Project documentation
  • Build logs & creative output
  • Knowledge notes

These represent real intellectual work that happened once. V5 has no mechanism to regenerate them.

The systems argument: V5 has a native learning flywheel — sessions generate observations, observations improve the model, better model means better effectiveness, better effectiveness means more sessions. Early in V5's life this loop is fragile. Bulk-porting V4 observations floods the pool with content calibrated for a different system. V5's own signal gets drowned out. The V4 observations become the dominant model — and they're wrong for V5.

Same-taxonomy trap

If V4 and V5 share a taxonomy (both have TELOS files with similar names), classification-based migration tools will fail. They try to reclassify already-correct content and produce low confidence with misrouting. Use direct diff-and-merge instead: read both versions, identify what V5 is missing, merge the delta.

Two-tier file pattern

V5 files loaded at startup need to stay concise. V4 depth goes below a ## Deep Context heading:

## Challenge Title

C0: **One-paragraph summary.** Quick-scan friendly.

## Deep Context

### Full depth here
Detailed analysis, source material, full context...

Startup gets the concise scan. Skills that read on-demand get the depth. Both live in one file.

Cluster knowledge notes

When triaging closeout artifacts (decisions, discoveries), group KEEP entries by theme into ~5-10 clustered notes rather than one file per entry. BM25 search retrieves one well-tagged file better than many tiny files with weak individual matches. Use V5-compliant frontmatter with status: evergreen and descriptive tags.

Conventions are capabilities

V4 artifacts with a CONVENTION.md or routing rule aren't just data — they're capabilities. Porting the data file is mechanical. Recreating the routing convention is a design task. Flag these for the user rather than silently copying the data and ignoring the convention.

Numbers from a real migration

MetricValue
V4 archive size~2.9GB, 408 work sessions
Manifest intents64 (consolidated from 94)
Content entering V5's active path~220 files + 433MB local search corpus

The ~1.4% ratio is correct, not a failure. Most V4 content is stock (V5 rebuilds natively), stale (outdated findings), or historical (session logs from a different system). The V4 backup remains accessible for reference — not porting is not losing.

Manifest blind spots V5 should watch for

V5's default bias is to DROP V4 capabilities

When assessing V4-unique skills, hooks, or agents against V5 equivalents, the receiving instance compares frontmatter and names, concludes "V5 already has this," and recommends DROP. On one real migration this was wrong ~80% of the time. Force deep cross-comparison: spawn agents to read the full content of both V4 and V5 versions side by side. The picture changed from 80% DROP to 10% DROP — V4 capabilities had depth, methodology, and features that V5's similarly-named equivalents didn't cover. Surface comparison is not comparison.

No stock/flow distinction

The manifest catalogs by intent, not by regenerability. V5 must classify stock vs. flow during triage.

No file-size warnings

Large archives (hundreds of MB) aren't flagged. V5 discovers the size at port time. Check du -sh before committing.

No retrieval-path context

The manifest describes V4's structure. V5 needs to map its own retrieval architecture before routing content.

Conventions aren't flagged as capabilities

CONVENTION.md files represent routing rules, not just data. The manifest treats them as files.

Starter Kit

Quick start

  1. Copy the template/ directory into your V4 installation
  2. Customize validate-manifest.ts: update VALID_CATEGORIES if yours differ, update the V5 regex to match your V5's naming conventions
  3. Open a fresh PAI session and paste the contents of BRIEF.md
  4. Let your PAI run the 2-pass process
  5. Review the generated PORT_MANIFEST.md yourself — this guide tells you what to look for
This process is not perfect

The methodology was tested on one real migration and refined through three iterations. It works, but it has blind spots (see Limitations). Your PAI may find new gotchas. If it does, the intent list and validator give you structured places to catch them — which is the whole point of having a process rather than winging it.
